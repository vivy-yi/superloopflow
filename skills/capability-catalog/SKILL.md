# Capability Catalog (v3.0)

The **Capability Catalog** is the runtime inventory of "what can my toolchain do right now?"

## Why a Catalog?

A typical Claude Code setup has 22+ plugins with 200+ capabilities. Without a catalog, you don't know:
- Which capabilities your toolchain provides
- Which LOOP phases have 0 providers
- Which capabilities have multiple providers
- How to choose between competing providers

## The Three Layers

### Layer 1: Must-have capabilities (~10)

These are the capabilities every AI-assisted development project SHOULD have:

| Capability | Why Must-Have | Default Provider |
|-----------|---------------|------------------|
| requirements-clarification | Without it, agents guess | superpowers/brainstorming |
| plan-decomposition | Without it, agents improvise | superpowers/writing-plans |
| spec-authoring | Without it, work is undocumented | planning-with-files/prd |
| implementation (any) | Core function | omc/autopilot OR superpowers/subagent |
| code-review | Without it, bugs ship | ecc/code-review |
| test-driven-development | Without it, regressions | superpowers/tdd |
| verification-before-completion | Without it, "done" is wishful thinking | superpowers/verification |
| continuous-learning | Without it, mistakes repeat | ecc/continuous-learning-v2 |
| meta-doc-authoring | Without it, knowledge decays | superloopflow/meta-doc-author |
| tool-inventory | Without it, plugins go stale | superloopflow/tool-inventory-audit |

### Layer 2: Domain-specific (200+)

Auto-inferred from plugin names and command/skill descriptions. They vary by installed plugins.

Run `/superloopflow:discover` to see what your specific environment provides. In a typical 22-plugin setup:

| Category | Examples |
|----------|----------|
| Backend | cqrs, event-store, saga-orchestration, microservices-patterns |
| Frontend | liquid-glass-design, frontend-patterns, react-patterns |
| LLM applications | cost-aware-llm-pipeline, eval-harness, rag-implementation |
| Security | security-review, hipaa-compliance |
| DevOps | docker-patterns, deployment-patterns, kubernetes-patterns |
| Data | postgres-patterns, databricks |
| Mobile | swift-concurrency, kotlin-coroutines |

### Layer 3: Project-specific (N items)

Each project has its own domain. Declared in `pipeline_defs/<pipeline>.yaml`:

```yaml
pipeline: video-pipeline
stages:
  - id: tts
    skill: providers/elevenlabs
    produces: audio
  - id: image-gen
    skill: providers/flux
    produces: images
  - id: video-compose
    skill: stages/ffmpeg-compose
    produces: video
```

## How to Use the Catalog

```bash
# 1. See full catalog
/superloopflow:discover

# 2. Find a specific capability
/superloopflow:capability code-review

# 3. Detect gaps (output flags phases with 0 providers)
/superloopflow:capability implementation-autopilot
```

## Discovery Protocol (v0.4 Read-only)

```bash
# Read-only: NEVER write to ~/.claude/plugins/
for plugin in $(jq -r '.enabledPlugins | keys[]' ~/.claude/settings.json); do
  # Read plugin.json
  cat ~/.claude/plugins/marketplaces/$plugin/plugin.json

  # Read commands
  ls ~/.claude/plugins/marketplaces/$plugin/commands/

  # Read skills
  ls ~/.claude/plugins/marketplaces/$plugin/skills/
done

# Write catalog to project
cat > .loop-state/capability-registry.yaml <<'EOF'
# Auto-generated by /superloopflow:discover
# DO NOT EDIT - re-run /superloopflow:discover to update
EOF
```

## When to Update the Catalog

The catalog is **dynamically generated** (Layer 2) or **statically declared** (Layer 3). It updates when:
- A new plugin is installed (re-run `/superloopflow:discover`)
- A plugin is updated (re-run discovery)
- A project's pipeline manifest changes (Layer 3)

There is **no manual catalog file to maintain**. The catalog is always derived from:
1. `~/.claude/settings.json` enabledPlugins
2. Each plugin's `plugin.json`, `commands/*.md`, `skills/*/SKILL.md` frontmatter
3. The project's `pipeline_defs/<pipeline>.yaml`

## Why Three Layers

| Need | Use layer |
|------|-----------|
| "What's the must-have checklist for AI dev?" | Layer 1 |
| "What can my current toolchain do?" | Layer 2 (via `/superloopflow:discover`) |
| "What does THIS project need?" | Layer 3 (in pipeline manifest) |

## Capability Registry Schema

```yaml
# .loop-state/capability-registry.yaml
version: "3.0"
generated: "2026-06-29T12:00:00Z"

must_have:
  requirements-clarification:
    provider: superpowers
    skill: brainstorming
    phase: 1
  plan-decomposition:
    provider: superpowers
    skill: writing-plans
    phase: 3
  implementation-autopilot:
    provider: omc
    command: /omc:autopilot
    phase: 5
  code-review:
    provider: ecc
    command: /ecc:code-review
    phase: 6

domain_specific:
  # Auto-inferred from plugins
  rag-implementation:
    provider: llm-application-dev
    skill: rag-implementation
    phase: 5
  react-patterns:
    provider: frontend-design
    skill: react-patterns
    phase: 5

project_specific:
  # From pipeline_defs/<pipeline>.yaml
  tts-synthesis:
    provider: local
    skill: providers/elevenlabs
    phase: 5
```

## Detecting Gaps

When a phase has 0 providers:

| Phase | Gap | Suggestion |
|-------|-----|------------|
| 1 (brainstorming) | requirements-clarification | Install superpowers or omc |
| 3 (planning) | plan-decomposition | Install superpowers |
| 5 (implementation) | implementation-autopilot | Install omc |
| 6 (review) | code-review | Install ecc |
| 7 (learning) | continuous-learning | Install ecc |
